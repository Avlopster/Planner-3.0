# Документация: Планировщик ресурсов отдела (Planner.py)

## 1. Общее описание

**Planner.py** — веб-приложение на Streamlit для планирования загрузки сотрудников и управления проектами. Использует SQLite в качестве хранилища данных и Plotly для визуализации.

### Технологический стек
- **Streamlit** — веб-интерфейс
- **SQLite** — база данных (файл `resource_planner.db`)
- **Pandas** — работа с данными
- **Plotly** — диаграммы и графики

### Структура проекта (после рефакторинга)

- **Planner.py** — точка входа: инициализация БД (одно подключение на сессию в `st.session_state["_db_conn"]`), загрузка централизованных стилей из `.streamlit/styles.css`, меню, диспетчер разделов (вызов `app_pages.*.render(conn)`).
- **.streamlit/config.toml** — тема приложения: светлая и тёмная (primaryColor, backgroundColor, secondaryBackgroundColor, textColor, font). Переключение: Settings → Theme в интерфейсе Streamlit.
- **.streamlit/styles.css** — общие стили: выравнивание кнопок меню в сайдбаре, отступы между секциями; для широкоформатных мониторов — `max-width: 100%` у основного контейнера контента и уменьшенные горизонтальные/вертикальные отступы. Подключается в Planner.py один раз после `set_page_config`.
- **config.py** — конфигурация по умолчанию: путь к БД (всегда абсолютный, относительно каталога проекта; переменная окружения `PLANNER_DB_PATH` при необходимости), константы ёмкости, плейсхолдеры дат, статусы, MONTHS_RU и др.; фактические значения пользовательских настроек хранятся в таблице `config` в БД и редактируются в разделе «Конфигурация». Каталог логов задаётся функцией `get_logs_dir()` (при доступной записи — подкаталог `logs` рядом с приложением, иначе `%LOCALAPPDATA%\Planner\Logs`); константы `LOGS_DIR`, `DB_LOG_PATH`, `ACTIONS_LOG_PATH`, `ERRORS_LOG_PATH`.
- **utils/app_logging.py** — логирование приложения: инициализация при старте (`init_app_logging()` вызывается в Planner.py до подключения к БД), три лог-файла с ротацией (planner_db.log — операции БД; planner_actions.log — действия пользователя; planner_errors.log — ошибки и предупреждения с сообщениями, понятными пользователю). API: `get_db_logger()`, `get_actions_logger()`, `get_errors_logger()`, `log_user_facing_error()`, `record_error()` (маппинг исключений на пользовательские формулировки). Ротация: 2 МБ, 5 копий, UTF-8.
- **database.py** — подключение к SQLite, схема таблиц, миграции. Использует `get_db_logger()` для записей в planner_db.log.
- **repository.py** — загрузка данных (load_roles, load_employees, load_projects, load_phases и т.д.), CRUD, конфиг (get_config, set_config, load_capacity_config, get_active_statuses, get_gantt_active_statuses, get_date_placeholders). Функции get_employee_name и get_project_name приводят id к нативному int для совместимости с SQLite при передаче значений из pandas/numpy.
- **load_calculator.py** — расчёт загрузки по дням (employee_load_by_day, employee_load_by_day_batch для нескольких сотрудников за один проход), сводка по отделу (department_load_summary), перегрузки (overload_shortfall), источники загрузки, кандидаты на замену.
- **capacity.py** — годовая ёмкость проектов (annual_project_capacity).
- **excel_import.py** — ядро импорта этапов и назначений на этапы (import_phases_core, import_phase_assignments_core).
- **excel_import_ui.py** — шаблоны и импорт по сущностям для UI (download_template, import_employees, import_vacations и др.).
- **msproject_import.py** — парсинг XML Microsoft Project (MSPDI): метаданные проекта, задачи, ресурсы, назначения; маппинг в проект и этапы Planner; импорт назначений (Assignments + fallback по полю «Исполнитель» в ExtendedAttribute).
- **utils/date_utils.py** — парсинг дат (parse_date_from_excel), диапазоны дат (date_range_list), normalize_date.
- **utils/type_utils.py** — приведение типов из БД: safe_int (в т.ч. bytes), safe_date_series; используются в repository и UI.
- **utils/chart_theme.py** — единая палитра цветов (COLOR_PALETTE) и шаблон Plotly (CHART_TEMPLATE, apply_chart_theme); используется в дашборде, аналитике, календаре и диаграмме Ганта для согласованного вида графиков.
- **components.py** — UI-компоненты: выбор даты с русскими месяцами (ru_date_picker), единообразный заголовок страницы (page_header(icon, title)), кнопки меню (menu_button).
- **app_pages/** — страницы разделов приложения; каждая реализует `render(conn)`:
  - **dashboard.py** — Дашборд (KPI, проекты по статусам, распределение сотрудников, отделы, сроки, загрузка, проекты без состава, этапы по проектам; на широких экранах блоки размещены в двух колонках для экономии вертикального пространства).
  - **roles.py** — Роли
  - **employees.py** — Сотрудники
  - **vacations.py** — Отпуска
  - **projects.py** — Проекты и этапы
  - **import_excel.py** — Импорт из Excel (вкладки по сущностям + вкладка «MS Project (XML)»)
  - **analytics.py** — Аналитика (фильтры, KPI, средняя и детальная загрузка в двух колонках, дни перегрузки, экспорт Excel).
  - **gantt.py** — Диаграмма Ганта (фильтры, чекбоксы этапов в сетке 4 колонки, слайдер высоты диаграммы, timeline).
  - **calendar.py** — Календарь (вкладки: отпуска и проекты по сотрудникам; проекты vs отпуска — два timeline в двух колонках, конфликты на всю ширину).
  - **config_page.py** — Конфигурация
  - **clear_data.py** — Очистить данные

Запуск: `streamlit run Planner.py` или `make run`. Тесты: `pytest tests/ -v` или `make test`.

## 2. Модель данных

### 2.1. Таблица `roles` (Роли)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| name | TEXT | Название роли (уникальное) |

Роли используются для классификации сотрудников (например, Senior, Junior). Роли Senior и Junior создаются по умолчанию и не могут быть удалены.

### 2.2. Таблица `employees` (Сотрудники)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| name | TEXT | ФИО сотрудника |
| role_id | INTEGER | Ссылка на роль |
| department | TEXT | Отдел / команда (опционально) |
| default_load_percent | REAL | Ставка занятости по умолчанию, % (по умолчанию 100) |

Ставка занятости используется при расчёте общей ёмкости (человеко-дней) в аналитике.

### 2.3. Таблица `vacations` (Отпуска)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| employee_id | INTEGER | Ссылка на сотрудника |
| start_date | DATE | Дата начала отпуска |
| end_date | DATE | Дата окончания отпуска |

У одного сотрудника может быть несколько периодов отпуска.

### 2.4. Справочник `project_statuses` (Статусы проектов)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| code | TEXT | Код статуса (уникальный): planned, in_progress, completed, cancelled |
| name | TEXT | Название: Планируется, В работе, Завершён, Отменён |
| is_active | BOOLEAN | Участвует в «активных» для отчётов и годовой ёмкости |
| is_gantt_active | BOOLEAN | Показывать при фильтре «Только активные» на диаграмме Ганта |

Справочник заполняется при первом запуске; настройки «Активные статусы» и «Статусы для фильтра Ганта» в разделе «Конфигурация» задают, какие коды учитывать.

### 2.5. Таблица `projects` (Проекты)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| name | TEXT | Название проекта |
| client_company | TEXT | Компания клиента (опционально) |
| lead_id | INTEGER | ID ведущего специалиста |
| lead_load_percent | REAL | Загрузка ведущего в % (по умолчанию 50) |
| project_start | DATE | Дата начала проекта |
| project_end | DATE | Дата окончания проекта |
| auto_dates | BOOLEAN | Автоопределение дат по этапам (0/1) |
| status_id | INTEGER | Ссылка на project_statuses(id); по умолчанию «В работе» |

**Примечание:** При `auto_dates = 1` даты проекта вычисляются автоматически по минимуму/максимуму дат этапов. При загрузке проектов имя статуса подставляется через JOIN (поле `status_name` в репозитории).

### 2.6. Таблица `project_juniors` (Рядовые сотрудники проекта)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| project_id | INTEGER | Ссылка на проект |
| employee_id | INTEGER | Ссылка на сотрудника |
| load_percent | REAL | Загрузка в % (по умолчанию 100) |

Уникальность: пара (project_id, employee_id). У проекта может быть **несколько** рядовых сотрудников.

### 2.7. Таблица `project_phases` (Этапы проекта)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| project_id | INTEGER | Ссылка на проект |
| name | TEXT | Название этапа |
| phase_type | TEXT | Тип: «Обычная работа» или «Командировка» |
| start_date | DATE | Дата начала этапа |
| end_date | DATE | Дата окончания этапа |
| completion_percent | INTEGER | Готовность этапа, % (0–100, шаг 5; по умолчанию 0). Участвует в расчёте готовности проекта и в заливке на диаграмме Ганта. |

### 2.8. Таблица `phase_assignments` (Назначения на этапы)
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Первичный ключ |
| phase_id | INTEGER | Ссылка на этап |
| employee_id | INTEGER | Ссылка на сотрудника |
| load_percent | REAL | Загрузка в % |

Используется для **индивидуальных** назначений на этап. Если назначений нет — наследуется состав проекта (ведущий + рядовые с их процентами). Для этапа типа «Командировка» без назначений занятость считается 100% для участников.

### 2.9. Таблица `config` (Настройки)

| Поле | Тип | Описание |
|------|-----|----------|
| key | TEXT | Ключ настройки (первичный ключ) |
| value | TEXT | Значение (строка) |

Хранятся пользовательские настройки: параметры годовой ёмкости (annual_working_days, typical_project_days, typical_lead_share, max_concurrent_projects), привязка ролей (capacity_role_junior_ids, capacity_role_senior_ids), annual_vacation_days, date_placeholder_start, date_placeholder_end, active_statuses, gantt_active_statuses. Редактируются в разделе «Конфигурация»; при отсутствии ключа в коде используются значения по умолчанию из `config.py`.

## 3. Функциональные модули

### 3.1. Роли (раздел «Роли»)
- **Добавление роли** — ввод названия и сохранение.
- **Редактирование роли** — изменение названия (кроме Senior/Junior).
- **Удаление роли** — при условии, что роль не назначена сотрудникам.
- Просмотр количества сотрудников по ролям.

### 3.2. Сотрудники (раздел «Сотрудники»)
- **Добавление сотрудника** — ФИО, выбор роли, отдел (опционально), ставка занятости в % (по умолчанию 100).
- **Редактирование** — изменение ФИО, роли, отдела и ставки.
- **Удаление** — при условии, что сотрудник не назначен на проекты или этапы.
- **Поиск** — по ФИО, роли или отделу.
- Отображение количества периодов отпуска.

### 3.3. Отпуска (раздел «Отпуска»)
- **Добавление отпуска** — выбор сотрудника, даты начала и окончания (русский выборщик дат). При сохранении выполняется **проверка пересечения** с существующими отпусками этого сотрудника; при пересечении выводится ошибка с датами конфликтующего отпуска.
- **Редактирование** — изменение дат периода отпуска (с той же проверкой пересечения, кроме редактируемой записи).
- **Удаление** — удаление записи об отпуске.
- Отображение всех периодов отпусков с привязкой к сотруднику.

### 3.4. Проекты и этапы (раздел «Проекты и этапы»)

#### Создание проекта
- Название, компания клиента.
- Ведущий специалист и загрузка ведущего (%).
- **Статус проекта** — Планируется, В работе, Завершён, Отменён (по умолчанию «В работе»).
- Опция «Автоопределение дат по этапам» — при включении даты проекта будут вычислены автоматически по этапам.
- Ручные даты начала/окончания (если автоопределение выключено).
- Несколько рядовых сотрудников с индивидуальной загрузкой (%).

#### Редактирование проекта
- Изменение всех параметров проекта, включая статус (выпадающий список из справочника `project_statuses`), состав и загрузку.
- **Сохранение статуса** — отдельная кнопка «Сохранить статус» рядом с выпадающим списком; после сохранения изменения видны при «Обновить данные» и на дашборде (используется одно подключение к БД на сессию).
- При включённом `auto_dates` даты проекта синхронизируются с min/max дат этапов с помощью служебной функции обновления дат проекта.
- **Поиск** по списку проектов — по названию, компании клиента или статусу.

#### Этапы проекта
- **Добавление этапа** — название, тип («Обычная работа» или «Командировка»), даты, **Готовность этапа, %** (выпадающий список 0, 5, 10, …, 100; по умолчанию 0).
- **Редактирование этапа** — название, тип, даты, **Готовность этапа, %** (выпадающий список с шагом 5 %). При открытии формы редактирования поля назначений инициализируются текущими данными этапа, поэтому изменение только названия, дат или типа не сбрасывает индивидуальные назначения.
- В списке этапов отображается колонка **Готовность %** (текущее значение по каждому этапу). Значение участвует в расчёте готовности проекта (взвешенная по длительности этапов) и в заливке шкалы проекта и этапов на диаграмме Ганта.
- **Индивидуальные назначения на этап** — при редактировании можно включить режим индивидуальных назначений, выбрать нескольких сотрудников и задать для каждого процент загрузки; чекбокс и список сотрудников по умолчанию заполняются ранее назначенными ресурсами; при сохранении старые назначения этапа очищаются и перезаписываются введёнными данными.
- **Удаление этапа** — каскадное удаление назначений.

### 3.5. Импорт из Excel (раздел «Импорт из Excel»)

Поддерживаются отдельные вкладки и шаблоны:

| Сущность | Лист Excel | Колонки |
|----------|------------|---------|
| Сотрудники | Сотрудники | ФИО, Роль, Отдел, Ставка % |
| Отпуска | Отпуска | Сотрудник, Дата начала, Дата окончания |
| Проекты | Проекты | Название проекта, Компания клиента, Ведущий, Загрузка ведущего %, Дата начала, Дата окончания, Автоопределение дат |
| Рядовые | Рядовые | Проект, Сотрудник, Загрузка % |
| Этапы | Этапы | Проект, Название этапа, Тип этапа, Дата начала, Дата окончания |
| Назначения на этапы | Назначения на этапы | Проект, Название этапа, Сотрудник, Загрузка % |

При импорте отпусков выполняется **проверка пересечения** периодов: при пересечении с существующим отпуском того же сотрудника запись не импортируется, в отчёт об ошибках выводится сообщение с датами конфликтующего отпуска.

Для каждого типа доступна кнопка «Шаблон» для скачивания пустого шаблона.

**Импорт из MS Project (XML)** — вкладка «MS Project (XML)» в разделе «Импорт из Excel»:
- Загрузка файла XML, экспортированного из Microsoft Project (формат обмена MSPDI).
- Опция «Импортировать только листовые задачи (Summary=0)» — при включении в этапы попадают только задачи без подзадач.
- Предпросмотр: название проекта, даты, таблица этапов (название, даты, % выполнения).
- Кнопка «Импортировать» создаёт один проект (название из Title или первой задачи уровня 0), все отобранные этапы со сроками и completion_percent, затем назначения на этапы: из блока Assignments (TaskUID, ResourceUID, Units) при наличии реальных ресурсов в XML; при их отсутствии — из расширенного атрибута «Исполнитель» (FieldID 188743731) у задач (строка имён через запятую). Тип этапа по умолчанию — «Обычная работа». Сотрудники для назначений подбираются по имени из справочника сотрудников Planner.

### 3.6. Дашборд (раздел «Дашборд»)

Обзорная страница с ключевыми метриками и визуализациями (без изменения данных).

- **KPI по статусам проектов** — метрики: количество проектов «В работе», «Планируется», «Завершён», «Отменён», всего проектов.
- **Диаграмма по статусам** — круговая диаграмма (Plotly) распределения проектов по статусам.
- **Распределение сотрудников по проектам** — таблица: проект, статус, ведущий, рядовые (ФИО), всего человек; горизонтальная столбчатая диаграмма «количество сотрудников по проектам».
- **Участие отделов в проектах** — при заполненном поле «Отдел» у сотрудников отображается stacked bar: сотрудники отделов по проектам.
- **Ближайшие сроки окончания проектов** — таблица проектов со статусом «В работе» или «Планируется», у которых дата окончания в ближайшие 30 дней (проект, дата окончания, ведущий).
- **Загрузка (сводка на 30 дней)** — сводка за период «сегодня + 30 дней»: всего человеко-дней, использовано, свободно (из `department_load_summary`); **«проектов можно взять»** берётся из **годовой ёмкости** (`annual_project_capacity`): макс. число проектов в год по составу Junior/Senior и отпускам минус активные проекты в текущем году; при нехватке ресурсов (нет одной из ролей) — предупреждение.
- **Блок «Прогноз»** — перед expander’ом «Прогноз: если добавить сотрудников» выводится явное уведомление: при нехватке специалистов для взятия ещё одного проекта — «Для взятия новых или реализации текущих проектов не хватает: X младших, Y старших специалистов» (значения из `annual_project_capacity`: `missing_juniors`, `missing_seniors`); при наличии перегрузок за 30 дней — «В выбранном периоде из-за перегрузок эквивалентно не хватает Z сотрудников» (из `overload_shortfall`). Далее — expander с сценариями «+1 младший» / «+1 старший» и пояснением об ограничении по младшим или старшим.
- **Проекты без полного состава** — для активных проектов («В работе», «Планируется») таблица проектов без ведущего или без рядовых сотрудников.
- **Этапы по проектам** — столбчатая диаграмма количества этапов по каждому проекту (данные из `project_phases`).

При отсутствии проектов выводится информационное сообщение с предложением добавить проекты в разделе «Проекты и этапы».

### 3.7. Аналитика (раздел «Аналитика»)

- **Период анализа** — выбор начала и конца периода.
- **Сводные метрики:**
  - Всего человеко-дней (ёмкость) — учитывается **ставка занятости** каждого сотрудника (`default_load_percent`).
  - Использовано дней.
  - Свободно дней.
  - **Количество проектов, которые можно взять** — по **годовой ёмкости** (`annual_project_capacity`): N = min(N_junior, N_senior) с учётом эффективного фонда F = 247 − 28 дней, типовой длительности проекта T = 30 дней, доли старшего x = 0.25 и лимита одновременных проектов; из N вычитаются активные проекты в текущем году (статусы «В работе», «Планируется»).
  - Предупреждение о нехватке сотрудников (если годовая ёмкость = 0 из-за отсутствия Junior или Senior).
- **Средняя загрузка сотрудников** — столбчатая диаграмма по сотрудникам.
- **Детальная загрузка сотрудника** — линейный график загрузки по дням.
- **Дни перегрузки (>100%)** — таблица сотрудник/дата/загрузка.
- **Блок «Прогноз»** — явное уведомление о недостающих специалистах (X младших, Y старших) из `annual_project_capacity` (`missing_juniors`, `missing_seniors`); при перегрузках за выбранный период — эквивалент недостающих сотрудников из `overload_shortfall(start_str, end_str)`. Затем expander «Прогноз: если добавить сотрудников» с сценариями найма.
- **Экспорт отчёта в Excel** — кнопка «Скачать отчёт (Excel)»: файл содержит листы «Сводка», «Загрузка по сотрудникам», «Дни перегрузки». На листе «Сводка» помимо метрик ёмкости и нехватки сотрудников выводятся: **Не хватает младших**, **Не хватает старших**, **Эквивалент недостающих (перегрузки)**.

### 3.8. Диаграмма Ганта (раздел «Диаграмма Ганта»)

- Отображение проектов и их этапов на временной шкале.
- Цветовое разделение по типу (Проект, Обычная работа, Командировка); тип **Выполнено** — заливка готовности на шкале проекта (зелёный отрезок).
- **Заливка готовности проекта:** основная шкала проекта (бар от начала до конца) частично закрашивается по проценту выполнения. Если у этапов задано поле **Готовность этапа, %** (`completion_percent`), процент проекта считается как взвешенная по длительности этапов сумма этих процентов; иначе — по текущей дате (этап полностью выполнен, если дата окончания ≤ текущей даты; частично — если текущая дата в интервале этапа). Длина заливки пропорциональна длительности всей шкалы проекта. При отсутствии этапов заливка не показывается.
- **Заливка готовности этапа:** для каждого этапа с «Готовность этапа, %» > 0 на той же строке диаграммы отображается зелёный отрезок «Выполнено» от начала этапа до позиции, пропорциональной этому проценту (аналогично заливке шкалы проекта).
- Даты проектов с автоопределением вычисляются по этапам.
- **Фильтр «Только активные проекты»** — чекбокс «Только активные проекты (В работе)»: при включении на диаграмму попадают только проекты со статусами из настройки «Статусы для фильтра Ганта» (задаётся в разделе «Конфигурация»; по умолчанию «В работе»).
- **Мультиселект «Проекты на диаграмме (выделенные отображаются)»** — выбор проектов для отображения; по умолчанию выбраны все (или все активные при включённом фильтре). На графике отображаются только выбранные проекты и их этапы.
- **Высота диаграммы** — вычисляется как `max(400, 32 * n_rows)`, где `n_rows` — число строк (проекты + этапы). Диаграмма расширяется вниз при любом количестве проектов и этапов; страница прокручивается, все выбранные элементы помещаются на график.
- Глобальный переключатель «Показывать этапы проектов (по умолчанию)» для управления тем, развёрнуты ли этапы по умолчанию.
- Кнопки «Развернуть этапы всех проектов» и «Свернуть этапы всех проектов», которые массово переключают состояния всех проектов и перерисовывают диаграмму.
- Отдельные чекбоксы для каждого проекта, позволяющие показать или скрыть его этапы на диаграмме независимо от других проектов.
- На оси задач верхний уровень занимают строки проектов, а под ними — строки этапов с указанием названия проекта и этапа; порядок строк строго соответствует порядку формирования списка задач (сначала проект, затем его этапы).

### 3.9. Календарь (раздел «Календарь»)

- Timeline отпусков и проектов по сотрудникам.
- Выбор периода отображения.
- Отпуска и проекты (ведущий и рядовые) отображаются в виде полос.
- Для проектов используются фактические даты проекта (с учётом `auto_dates`, если включено).
- События фильтруются по выбранному периоду: в календарь попадают только события, пересекающиеся с заданным интервалом.

### 3.10. Конфигурация (раздел «Конфигурация»)

Все настройки приложения редактируются через интерфейс (одна закладка меню «Конфигурация»).

- **Параметры формулы ёмкости и привязка ролей** — рабочие дни в году (D), типовая длительность проекта (T), доля старшего на проект (x), макс. одновременных проектов; выбор ролей для подсчёта n_junior и n_senior (мультиселект по справочнику ролей). Сохранение в таблицу `config`.
- **Отпуска по умолчанию** — число рабочих дней отпуска в году по умолчанию (используется при расчёте эффективного фонда, когда в системе ещё нет записей об отпусках).
- **Плейсхолдеры дат** — даты начала и окончания по умолчанию для невалидных или пустых дат при импорте из Excel и при создании проектов с автоопределением дат. Формат YYYY-MM-DD; при сохранении выполняется проверка.
- **Статусы проектов** — два поля (ввод через запятую): «Активные статусы» (проекты с этими статусами учитываются в отчётах и годовой ёмкости) и «Статусы для фильтра Ганта» (какие проекты показывать при включённом чекбоксе «Только активные проекты» на диаграмме Ганта). На странице отображаются текущие уникальные статусы из проектов для справки.
- **О системе** — путь к БД (read-only): всегда абсолютный путь; по умолчанию — каталог проекта + `resource_planner.db`; при заданной переменной окружения `PLANNER_DB_PATH` используется она (относительный путь дополняется каталогом проекта). JSON с текущими значениями всех настроек.

После сохранения в любом блоке выполняется сброс кеша и обновление страницы.

### 3.11. Очистить данные (раздел «Очистить данные»)

- Удаление всех данных: сотрудников, проектов, этапов, отпусков, ролей (кроме Senior и Junior).
- Требует подтверждения.

## 4. Расчёт загрузки

### Алгоритм `employee_load_by_day(employee_id, start_date, end_date)`

Для каждого дня в периоде:

1. **Отпуск** — в дни отпуска загрузка не начисляется.
2. **Проекты** — для каждого проекта, в котором участвует сотрудник (как ведущий или рядовой):
   - Определяются границы проекта: при `auto_dates` — min/max дат этапов, иначе — project_start, project_end.
   - Для каждого дня в границах проекта:
     - Если день попадает в этап:
       - Если у этапа есть `phase_assignments` — берётся загрузка из назначения для данного сотрудника.
       - Иначе: для этапа «Командировка» — 100%; для «Обычная работа» — наследуется загрузка из проекта (lead_load или junior_load).
     - Если день не в этапе — наследуется загрузка из проекта.
3. Загрузка суммируется по всем проектам. При превышении 100% в аналитике используется clip(upper=1.0) для подсчёта использованных дней.

### Ёмкость и сводка по периоду (`department_load_summary`)

- **Общая ёмкость (человеко-дней)** считается с учётом **ставки занятости** каждого сотрудника (`default_load_percent`): для каждого сотрудника к ёмкости добавляется `(default_load_percent / 100) × количество дней в периоде`.
- Использованные дни — сумма по всем сотрудникам загрузки по дням (clip не более 1.0 за день).
- Свободные дни = ёмкость − использованные дни. В интерфейсе для метрики **«проектов можно взять»** и предупреждения о нехватке сотрудников используется не эта сводка, а результат функции годовой ёмкости (см. ниже).

### Годовая ёмкость проектов (`annual_project_capacity`)

- **Назначение:** расчёт максимального числа проектов в год по документированной формуле с учётом состава отдела (роли Junior и Senior), отпусков и ограничения одновременных проектов.
- **Параметры:** берутся из таблицы `config` (раздел «Конфигурация»): рабочие дни в году D, отпуск по умолчанию V (при отсутствии отпусков в БД), эффективный фонд F = D − V; типовая длительность проекта T, доля старшего x, макс. одновременных проектов; привязка ролей для n_junior/n_senior. При отсутствии ключа в БД используются значения по умолчанию из `config.py` (D=247, V=28, T=30, x=0.25 и т.д.).
- **Формула:** циклов в году = ⌊F/T⌋; N_junior = n_junior × циклов; P = min(max_concurrent, ⌊n_senior/x⌋); N_senior = P × циклов; N_capacity = min(N_junior, N_senior). Активные проекты в году (N_active) — проекты со статусом «В работе» или «Планируется», пересекающиеся с календарным годом. **Проектов можно взять** = max(0, N_capacity − N_active).
- **Возвращаемые поля:** кроме N_capacity, N_active, projects_possible, missing_employees — **missing_juniors** и **missing_seniors** (сколько не хватает младших/старших для возможности взять ещё один проект в году; см. logic.md, п. 6.2).
- **Где используется:** метрика «Проектов можно взять» на дашборде и в разделе «Аналитика», явное уведомление в блоке «Прогноз», экспорт сводки в Excel.

### Недостающее количество людей из-за перегрузок (`overload_shortfall`)

- **Функция:** `overload_shortfall(start_date, end_date)`.
- **Назначение:** по дням с загрузкой > 100 % за период оценить эквивалент недостающих FTE (сумма избытков (load − 1.0) по всем сотрудникам и дням, делённая на число дней в периоде).
- **Возвращает:** `total_overload_ftedays`, `days_in_period`, `equivalent_missing_fte`.
- **Где используется:** блок «Прогноз» на дашборде (период 30 дней) и на странице «Аналитика» (выбранный период); лист «Сводка» при экспорте в Excel.

## 5. Миграции и совместимость

- При первом запуске создаются все таблицы.
- Миграция старых колонок `projects` (senior_id, junior_id и т.д.) в новую схему (lead_id, lead_load_percent, project_juniors); при миграции плейсхолдеры дат берутся из таблицы `config` (date_placeholder_start, date_placeholder_end).
- Добавление колонок при их отсутствии: в `projects` — `status` (TEXT, по умолчанию «В работе»), затем `status_id` (ссылка на `project_statuses`); в `employees` — `department`, `default_load_percent` (по умолчанию 100).
- Справочник `project_statuses`: создаётся при первом запуске и заполняется кодами (planned, in_progress, completed, cancelled). Миграция данных из колонки `projects.status` в `projects.status_id` с последующим удалением колонки `status`.
- Таблица `config`: при первом запуске заполняется ключами по умолчанию (ёмкость, роли, date_placeholder_start/end, active_statuses, gantt_active_statuses); для существующих БД миграция добавляет недостающие ключи.
- Перенос данных из таблицы `business_trips` в `project_phases` с типом «Командировка».
- Перенос отпусков из колонок `vacations` в таблице `employees` в отдельную таблицу `vacations`.

## 6. Интерфейс

- **Боковое меню** — переключение между разделами: Дашборд, Роли, Сотрудники, Отпуска, Проекты и этапы, Импорт из Excel, Аналитика, Диаграмма Ганта, Календарь, Конфигурация, Очистить данные. Стили кнопок меню задаются в `.streamlit/styles.css`.
- **Тема и оформление** — тема приложения настраивается в `.streamlit/config.toml` (светлая и тёмная); переключение: меню Streamlit → Settings → Theme. Общие стили (сайдбар, отступы между секциями) в `.streamlit/styles.css`, подключаются один раз в Planner.py. Графики Plotly используют единую палитру и шаблон из `utils/chart_theme.py`.
- **Структура страниц** — на дашборде и в аналитике блоки KPI и фильтров обёрнуты в контейнеры, секции разделены `st.divider()`; заголовки страниц через `page_header(icon, title)` из components. После экспорта отчёта в Excel в аналитике показывается тост «Отчёт сохранён».
- **Выбор дат** — компонент с русскими названиями месяцев (Январь–Декабрь), выбор дня, месяца, года.
- **Подключение к БД** — одно подключение на сессию (`st.session_state["_db_conn"]`), чтобы после сохранения данных (в т.ч. смена статуса проекта) кнопка «Обновить данные» и дашборд видели актуальные данные без расхождений из-за разных подключений или путей к файлу БД.
- Кеширование загрузки данных с TTL 60 секунд (`@st.cache_data(ttl=60)`).

**План доработки интерфейса и процесс:** пошаговый тасклист — [New_interface.md](New_interface.md); правила выполнения работ по тасклисту — [docs/workflow.md](docs/workflow.md).

## 7. Запуск и установка

**Разработка (из каталога Planner 3.0):**
```bash
streamlit run Planner.py
```
По умолчанию приложение доступно по адресу `http://localhost:8501`.

**Логи приложения:** при работе ведутся три лог-файла (см. раздел «Логи приложения» в корневом README): planner_db.log, planner_actions.log, planner_errors.log — в каталоге `LOGS_DIR` (подкаталог `logs` рядом с приложением или `%LOCALAPPDATA%\Planner\Logs`). При обращении в поддержку рекомендуется приложить planner_errors.log.

**Установка на Windows для конечных пользователей:** в каталоге **Planner 3.0** подготовлен установщик Inno Setup: каталог `installer/` со скриптами сборки дистрибутива (`build_dist.ps1`, `download_python_embedded.ps1`, `build_installer.ps1`) и сценарий `Planner.iss`. Версия установщика задаётся в `Planner.iss` константой `MyAppVersion`; она подставляется в имя приложения, версию и в свойства exe. Готовый установщик `PlannerSetup.exe` разворачивает приложение с встроенным Python, создаёт ярлык на рабочем столе (файл .url на http://localhost:8501) и при выборе — регистрирует Planner как службу Windows для автозапуска (подробнее см. [installer/README.md](installer/README.md)). Инструкция для пользователя — `installer/Инструкция_по_установке.txt`. Журнал установки и удаления записывается в папку `Logs` в каталоге установки приложения (файлы `install_*.log`, `uninstall_*.log`); при обращении в поддержку можно приложить последний лог установки и при необходимости **planner_errors.log** из каталога логов приложения.
