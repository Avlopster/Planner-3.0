# What's new

**4 марта 2026**

## Установщик с разностным обновлением

- **installer/** — в Planner 3.0 добавлен каталог установщика (Inno Setup) с поддержкой разностного обновления. При повторной установке: конфигурация Streamlit и БД не перезаписываются; venv обновляется только через `pip install --upgrade`; VC Redist и опция службы пропускаются.
- **build_dist.ps1** — адаптирован для Planner 3.0 (источник приложения — корень Planner 3.0, исключён каталог `installer` при копировании).
- **Проверка службы** — перед установкой проверяется, запущена ли служба Planner; предлагается остановить её или продолжить с предупреждением.
- **Запуск через службу** — опция «Start Planner now» (CMD) удалена; добавлен «Start Planner service now» (запуск службы без окна). При обновлении служба автоматически запускается по завершении.
- **Удалён Планировщик заданий** — автозапуск только через службу Windows.

## Система логирования приложения

- **Три лог-файла** — при работе приложения ведутся отдельные журналы: **planner_db.log** (подключение к БД, миграции, служебные сообщения СУБД), **planner_actions.log** (действия пользователя: создание/изменение/удаление ролей, сотрудников, проектов, этапов, отпусков, импорт, очистка данных), **planner_errors.log** (ошибки и предупреждения с формулировками, понятными пользователю и поддержке).
- **Расположение** — каталог логов: при наличии прав записи — подкаталог `logs` рядом с приложением; иначе — `%LOCALAPPDATA%\Planner\Logs`. Ротация: до 2 МБ на файл, до 5 резервных копий, кодировка UTF-8.
- **Обработчик ошибок** — модуль `utils/app_logging.py`: `init_app_logging()` (вызов при старте в Planner.py), `get_db_logger()`, `get_actions_logger()`, `get_errors_logger()`, `log_user_facing_error()`, `record_error()` с маппингом типичных исключений на пользовательские сообщения. Все сообщения об ошибках, показываемые в интерфейсе (st.error/st.warning), дублируются в planner_errors.log.
- **Конфигурация** — в `config.py` добавлены `get_logs_dir()`, `LOGS_DIR`, `DB_LOG_PATH`, `ACTIONS_LOG_PATH`, `ERRORS_LOG_PATH`. Документация: раздел «Логи приложения» в корневом README.md, раздел 7 в Planner.md.

## Импорт из MS Project (XML), исправления, установщик v2.1

- **Импорт из MS Project (XML)** — в разделе «Импорт из Excel» добавлена вкладка «MS Project (XML)». Загрузка файла XML (формат MSPDI), опция «только листовые задачи», предпросмотр проекта и этапов. Импорт создаёт один проект, этапы со сроками и % выполнения, назначения на этапы — из блока Assignments (при наличии ресурсов с именами в XML) или из расширенного атрибута «Исполнитель» у задач. Модуль `msproject_import.py`: парсинг метаданных, задач, ресурсов, назначений; маппинг в проект/этапы/назначения; тесты в `tests/test_msproject_import.py`.
- **Исправление sqlite3.InterfaceError на странице «Календарь»** — в `repository.get_employee_name` и `get_project_name` идентификаторы приводятся к нативному `int` перед запросом к БД; добавлена проверка на `None` и обработка исключений при приведении типа. Устраняет ошибку «bad parameter or other API misuse» при передаче значений из pandas/numpy (например при формировании кандидатов на замену в конфликтах «отпуск + проект»).
- **Установщик Windows — контроль версий** — в `installer/Planner.iss` введена константа `MyAppVersion` (старт с 2.1.0). Версия подставляется в AppVerName, AppVersion и в ресурсы exe (VersionInfoVersion и др.). Для нового релиза достаточно изменить одну строку в .iss.
- **Журнал установки и удаления** — установщик и деинсталлятор записывают действия в папку `Logs` в каталоге установки приложения (файлы `install_*.log`, `uninstall_*.log`). В лог попадают шаги: копирование файлов, VC Redist, создание venv, ярлык на рабочем столе, регистрация задачи в Планировщике заданий. При обращении в поддержку рекомендуется приложить последний лог установки или удаления.
- **Streamlit** — замена устаревшего параметра `use_container_width=True` у `st.dataframe` на `width="stretch"` в предпросмотре импорта MS Project.

## Сохранение статуса проекта и единое подключение к БД

- **Сохранение статуса проекта** — смена статуса в разделе «Проекты и этапы» (кнопка «Сохранить статус») теперь корректно сохраняется: после «Обновить данные» и на дашборде отображается новый статус.
- **Одно подключение к БД на сессию** — подключение к SQLite хранится в `st.session_state["_db_conn"]` и не пересоздаётся при каждом rerun; все разделы (Проекты, Дашборд и т.д.) работают с одним и тем же подключением, поэтому запись и чтение идут из одной и той же БД.
- **Путь к БД в config** — путь к файлу БД в `config.py` всегда абсолютный и вычисляется относительно каталога проекта (каталог `config.py`); при относительном значении переменной окружения `PLANNER_DB_PATH` он дополняется этим каталогом. Это исключает расхождение путей при разных текущих каталогах при запуске Streamlit.

## Конфигурация в отдельную закладку меню

- **Вся настройка через интерфейс** — все пользовательские параметры вынесены в одну закладку меню **«Конфигурация»** и редактируются через формы (без правки кода).
- **Блоки на странице «Конфигурация»:**
  - **Параметры формулы ёмкости и привязка ролей** — рабочие дни в году, типовая длительность проекта, доля старшего, макс. одновременных проектов; выбор ролей для n_junior и n_senior (как раньше).
  - **Отпуска по умолчанию** — годовое число рабочих дней отпуска по умолчанию (используется, когда в системе ещё нет записей об отпусках).
  - **Плейсхолдеры дат** — даты начала и окончания по умолчанию для невалидных/пустых дат при импорте и в проектах (формат YYYY-MM-DD, с проверкой при сохранении).
  - **Статусы проектов** — «Активные статусы» (для отчётов и годовой ёмкости) и «Статусы для фильтра Ганта» (при включённом «Только активные») задаются строкой через запятую; на странице показываются текущие уникальные статусы из проектов.
  - **О системе** — read-only путь к БД и JSON с текущими значениями конфигурации.
- **Хранение:** настройки сохраняются в таблице `config` (ключ–значение); `config.py` остаётся источником значений по умолчанию при отсутствии ключа в БД.
- **Использование в коде:** параметры читаются через `repository.get_config`, `repository.get_active_statuses`, `repository.get_gantt_active_statuses`, `repository.get_date_placeholders`; плейсхолдеры дат используются в миграциях БД, импорте Excel и при создании/редактировании проектов с автоопределением дат.
- **Документация** — обновлены Planner.md (модель данных: таблица config, раздел 3.10 «Конфигурация», меню, миграции), logic.md (п. 6.2 и 10.2 — конфигурация из БД), test.md (п. 8.1 — конфигурация через интерфейс), What's new.md.

## Готовность этапов и заливка на диаграмме Ганта

- **Поле готовности этапа** — в таблицу `project_phases` добавлена колонка `completion_percent` (0–100 %, по умолчанию 0). В разделе «Проекты и этапы» в списке этапов отображается колонка «Готовность %»; при добавлении и редактировании этапа доступен выпадающий список «Готовность этапа, %» с шагом 5 % (0, 5, 10, …, 100). Значение привязывается к этапу и сохраняется в БД.
- **Расчёт готовности проекта** — функция `gantt_project_completion_percent` учитывает `completion_percent` этапов: если колонка задана, готовность проекта считается как взвешенная по длительности этапов сумма этих процентов; иначе используется расчёт по текущей дате (как раньше). Готовность участвует в заливке шкалы проекта на диаграмме Ганта.
- **Заливка этапов на Ганте** — для каждого этапа с `completion_percent` > 0 на той же строке диаграммы рисуется зелёный отрезок «Выполнено» от начала этапа до позиции, пропорциональной проценту готовности этапа (аналогично заливке шкалы проекта).
- **Импорт из Excel** — при импорте этапов в БД для новых записей подставляется `completion_percent = 0`.
- **Документация** — обновлены Planner.md (модель данных 2.6, разделы 3.4 и 3.8), logic.md (п. 10.4 и новый п. 10.5), test.md (п. 8.1), What's new.md.

## Доработки за сегодня

- **Блок «Прогноз» — явное уведомление о недостающих специалистах**  
  На дашборде и в разделе «Аналитика загрузки» перед expander’ом «Прогноз: если добавить сотрудников» выводится сообщение: *«Для взятия новых или реализации текущих проектов не хватает: X младших, Y старших специалистов»*. Значения X и Y считаются по текущей формуле годовой ёмкости (поля `missing_juniors` и `missing_seniors` в `annual_project_capacity()`).

- **Недостающее количество людей из-за перегрузок**  
  Добавлена функция `overload_shortfall(start_date, end_date)`: по дням с загрузкой > 100 % считается сумма избыточной загрузки и эквивалент недостающих FTE за период (человеко-дни перегрузки / число дней).
