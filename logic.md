# Логика расчётов и проверок (Planner)

Документ описывает исключительно логику вычислений и проверок для функциональности приложения.

---

## 1. Проверка пересечения периодов отпусков

**Назначение:** запретить сохранение отпуска, если новый период пересекается с уже существующими отпусками того же сотрудника.

**Где используется:** добавление и редактирование отпуска (раздел «Отпуска»), импорт отпусков из Excel.

**Алгоритм (`check_vacation_overlap`):**

1. Загружаются все отпуска сотрудника по `employee_id`.
2. Даты нового периода и существующих отпусков приводятся к типу `date`.
3. Для каждого существующего отпуска (кроме исключённого при редактировании — `exclude_vacation_id`):
   - Проверяется условие пересечения интервалов:
     - **Пересечение есть**, если: `new_start <= existing_end` **и** `new_end >= existing_start`.
4. При первом же пересечении возвращается `(True, данные_конфликтующего_отпуска)`, иначе `(False, None)`.

**Исключение при редактировании:** при обновлении отпуска его собственный `id` передаётся в `exclude_vacation_id`, чтобы не считать конфликтом совпадение периода с самим собой.

## 2. Проверка конфликта «отпуск / проект»

**Назначение:** показать дни, когда сотрудник формально и в отпуске, и назначен на проект (одновременно).

**Где используется:** раздел «Календарь», вкладка «Проекты vs отпуска (два окна)», блок «Конфликты отпуск/проект».

**Задуманный алгоритм:**

1. Определяется список сотрудников для проверки (все или только участники выбранных проектов).
2. Для каждого сотрудника:
   - Берутся его отпуска из `df_vac`.
   - Считается загрузка по дням: `employee_load_by_day(emp_id, start_str, end_str)`.
3. Для каждого дня в периоде:
   - Проверяется, входит ли день в какой-либо отпуск сотрудника (`vac_start <= day_date <= vac_end`).
   - Берётся загрузка за этот день из `load_df`.
   - **Конфликт:** день одновременно в отпуске **и** загрузка за этот день > 0 — в отчёт добавляется строка (сотрудник, дата, загрузка, тип «Отпуск + проект»).
4. Результат выводится в виде таблицы (с удалением дубликатов).

**Реализация:** В `employee_load_by_day` добавлен параметр `ignore_vacations=False`. При `ignore_vacations=True` загрузка по проектам начисляется даже в дни отпуска — это позволяет корректно выявлять конфликты «отпуск + проект» (сотрудник формально назначен на проект и одновременно в отпуске).

## 3. Диапазон дат по дням

**Назначение:** получить список всех календарных дней между двумя датами включительно.

**Функция:** `date_range_list(start, end)`.

**Формула:**

- `delta = end - start` (разница в днях).
- Список: для каждого `i` от 0 до `delta.days`: день = `start + timedelta(days=i)`.
- В результат входят и `start`, и `end`.

## 4. Загрузка сотрудника по дням

**Назначение:** для каждого дня в заданном периоде вычислить суммарную загрузку сотрудника по проектам (доля ставки от 0 до 1 и выше при перегрузке).

**Функция:** `employee_load_by_day(employee_id, start_date, end_date)`.

**Входные данные:** проекты, этапы, назначения на этапы, рядовые по проектам, отпуска сотрудника.

**Батчевый вариант:** `employee_load_by_day_batch(conn, employee_ids, start_date, end_date)` — та же логика, но за один проход по проектам/этапам накапливается загрузка для всех переданных `employee_ids`. Возвращает DataFrame с колонками `employee_id`, `date`, `load`. Используется в `department_load_summary`, `overload_shortfall` и на странице «Аналитика» для ускорения при большом числе сотрудников.

**Алгоритм (одиночный и батч):**

1. Формируется таблица по дням: для каждого дня в `[start_date, end_date]` начальная загрузка = 0 (в батче — для каждой пары (employee_id, date)).
2. Отпуска сотрудника (или по каждому employee_id в батче) приводятся к списку пар (начало, конец) в виде дат.
3. Для каждого проекта:
   - **Границы проекта:**
     - Если у проекта включено автоопределение дат (`auto_dates`): начало = min дат начала этапов, конец = max дат окончания этапов.
     - Иначе: начало и конец берутся из `project_start`, `project_end`.
   - Определяется участие сотрудника в проекте: ведущий (с `lead_load_percent`) или рядовой (из `project_juniors` с `load_percent`). Если не участвует — проект пропускается.
   - Для каждого дня в границах проекта:
     - Если день попадает в любой отпуск сотрудника — загрузка не добавляется (`continue`).
     - Иначе ищется этап, в который попадает этот день (по датам этапа).
     - **Если день в этапе:**
       - Если у этапа есть индивидуальные назначения (`phase_assignments`): для этого сотрудника берётся его `load_percent` по этапу (в долях), добавляется к загрузке дня.
       - Если назначений нет: для этапа типа «Командировка» добавляется 1.0; для «Обычная работа» — суммы долей ведущего/рядовых по проекту (как определено выше).
     - **Если день не в этапе:** к загрузке дня добавляются доли ведущего/рядовых по проекту.
4. Загрузка по разным проектам суммируется; один день может дать больше 1.0 (перегрузка).

**Итог:** DataFrame с колонками `date` и `load` (число, доля ставки).

## 5. Обновление дат проекта по этапам

**Назначение:** пересчитать `project_start` и `project_end` у проекта по фактическим датам его этапов (для проектов с автоопределением дат).

**Функция:** `update_project_dates_from_phases(project_id)`.

**Алгоритм:**

1. Загружаются все этапы проекта (`project_phases` по `project_id`).
2. Если этапов нет — возвращается `False`, изменения не вносятся.
3. Иначе:
   - `project_start` = min дат начала этапов (`start_date`).
   - `project_end` = max дат окончания этапов (`end_date`).
4. В БД выполняется `UPDATE projects SET project_start = ?, project_end = ? WHERE id = ?`.

**Когда вызывается:** при добавлении/редактировании/удалении этапа, если у проекта включено автоопределение дат.

## 6. Сводка загрузки отдела и годовая ёмкость проектов

### 6.1. Сводка за период (человеко-дни)

**Функция:** `department_load_summary(start_date, end_date)`.

**Назначение:** за выбранный период оценить общую ёмкость отдела в человеко-днях, занятость и свободные дни. Метрика «Проектов можно взять» для отображения **не** берётся из этой функции (см. п. 6.2).

**Алгоритм:**

1. **Ёмкость:** для каждого сотрудника учитывается `default_load_percent`.  
   `total_capacity_days = сумма (default_load_percent/100 × число_дней_в_периоде)`.

2. **Занято:** вызывается один раз `employee_load_by_day_batch(conn, список_id_всех_сотрудников, start_date, end_date)`; по полученному DataFrame загрузка по дням ограничивается сверху 1.0 и суммируется.  
   `total_used` = сумма по всем строкам батча (или сумма по сотрудникам из того же батча).

3. **Свободно:**  
   `free_capacity = total_capacity_days - total_used`.

4. В возвращаемом словаре по-прежнему есть поля `projects_possible` и `missing_employees` (старая логика по свободным человеко-дням и 45 чел.-дням на проект), но в интерфейсе для метрики «Проектов можно взять» используется результат функции **годовой ёмкости** (п. 6.2).

**Возвращаемый словарь:**  
`total_capacity_days`, `used_days`, `free_days`, `projects_possible`, `missing_employees`.

### 6.2. Годовая ёмкость проектов («Проектов можно взять»)

**Назначение:** корректная оценка максимального числа проектов в год по составу отдела (младшие/старшие), отпускам и ограничению одновременных проектов — по документированной формуле.

**Функция:** `annual_project_capacity(year=None)`. Год по умолчанию — текущий.

**Параметры:** берутся из таблицы `config` в БД (раздел «Конфигурация»); при отсутствии ключа — значения по умолчанию из `config.py`: рабочие дни в году 247, отпуск по умолчанию 28, типовая длительность проекта 30, доля старшего 0.25, макс. одновременных проектов 4. Роли для n_junior/n_senior задаются в конфигурации (по умолчанию по имени «Junior»/«Senior»). Годовой отпуск V при пустых отпусках — из `repository.get_config(conn, 'annual_vacation_days', …)`.

**Алгоритм:**

1. **Эффективный фонд:** F = ANNUAL_WORKING_DAYS − V (V = 28 или из настроек).
2. **Состав:** n_junior = число сотрудников с ролью «Junior», n_senior = число с ролью «Senior».
3. **Циклов в году:** cycles = ⌊F / T⌋ (T = типичная длительность проекта, по умолчанию 30).
4. **По младшим:** N_junior = n_junior × cycles.
5. **По старшим:** P = min(MAX_CONCURRENT_PROJECTS, ⌊n_senior / x⌋), N_senior = P × cycles (x = доля старшего на проект, по умолчанию 0.25).
6. **Годовая ёмкость:** N_capacity = min(N_junior, N_senior) (если обе роли есть; иначе 0).
7. **Активные проекты в году:** N_active = число проектов, чей статус входит в список статусов для **расчёта ёмкости** (`repository.get_capacity_statuses(conn)`, флаг `project_statuses.is_capacity = 1`; по умолчанию «В работе»), и даты которых пересекаются с выбранным годом.
8. **Проектов можно взять:** projects_possible = max(0, N_capacity − N_active).

**Возвращаемый словарь:**  
`N_capacity`, `N_active`, `projects_possible`, `missing_employees` (1, если N_capacity = 0 из-за отсутствия одной из ролей), `missing_juniors`, `missing_seniors`.

**Недостающие младшие и старшие (`missing_juniors`, `missing_seniors`):**  
Показывают, сколько специалистов не хватает для возможности взять **ещё один** проект в году (по текущей формуле ёмкости).  
- **missing_juniors:** если ёмкость ограничена младшими (`capacity_limited_by == 'juniors'`): при cycles > 0 — max(0, ceil((N_active+1)/cycles) − n_junior); при cycles = 0 (длинные проекты) — max(0, N_active+1 − n_junior).  
- **missing_seniors:** если ограничение по старшим и P < max_concurrent_projects — max(0, ceil((P+1)·x) − n_senior).  
Эти значения выводятся в блоке «Прогноз» на дашборде и на странице «Аналитика загрузки» в виде явного уведомления: «Для взятия новых или реализации текущих проектов не хватает: X младших, Y старших специалистов.»

### 6.3. Недостающее количество людей из-за перегрузок (по периоду)

**Назначение:** по дням с загрузкой > 100 % оценить эквивалент недостающих FTE за выбранный период.

**Функции:** `overload_shortfall(start_date, end_date)`, `overloaded_employees_in_period(start_date, end_date)`.

**Алгоритм:**  
1. Вызывается один раз `employee_load_by_day_batch(conn, список_id_всех_сотрудников, start_date, end_date)`.  
2. По полученному DataFrame отбираются строки с load > 1.0; для них суммируется избыток (load − 1.0).  
3. **total_overload_ftedays** = эта сумма.  
4. **equivalent_missing_fte** = total_overload_ftedays / days_in_period (дней в периоде), округлённо.  

**Возвращаемые данные:**  
- `overload_shortfall`: `total_overload_ftedays`, `days_in_period`, `equivalent_missing_fte`.  
- `overloaded_employees_in_period`: список сотрудников с перегрузкой (`employee_id`, `employee_name`, `overload_days`, `overload_ftedays`, `avg_overload_percent`).

**Где используется:** на дашборде (период 30 дней) и на странице «Аналитика загрузки» (выбранный период) в блоках «Прогноз» и «Подсказка по оптимизации персонала»: выводится эквивалент нехватки и топ перегруженных сотрудников.

## 7. Удаление сущностей (условия)

**Сотрудник:** удалить можно только если он не является ведущим ни в одном проекте и не входит в `project_juniors`. Иначе выводится ошибка, удаление не выполняется.

**Роль:** удалить можно только если ни один сотрудник не ссылается на эту роль (`role_id`). Иначе — ошибка, удаление не выполняется.

**Проект, отпуск:** удаляются без дополнительных проверок (каскады заданы на уровне БД при необходимости).

## 8. Редактирование этапа — сохранение назначений

**Назначение:** при изменении имени, дат или типа этапа не сбрасывать индивидуальные назначения на этап (phase_assignments) в режим наследования от проекта.

**Где используется:** раздел «Проекты и этапы», форма редактирования этапа.

**Поведение:**

1. При открытии формы редактирования этапа поля «Назначения на этап» инициализируются текущими данными:
   - Чекбокс «Использовать индивидуальные назначения» включается, если у этапа уже есть записи в `phase_assignments`.
   - Мультиселект сотрудников заполняется списком назначенных сотрудников.
   - Для каждого сотрудника процент загрузки подставляется из `phase_assignments` (по умолчанию 100 %).
2. Если пользователь изменяет только название, даты или тип этапа и нажимает «Сохранить», введённые назначения (в т.ч. ранее существующие) сохраняются.
3. Переход к наследованию от проекта: снять чекбокс «Использовать индивидуальные назначения» или очистить список сотрудников — при сохранении назначения этапа удаляются, для этапа применяется наследование состава проекта.

## 9. Дашборд: агрегаты и фильтры

**Назначение:** обзорная страница с метриками по проектам, составу команд и срокам без изменения данных.

**Где используется:** раздел «Дашборд» (пункт в боковом меню).

**Используемые данные:** `load_projects()`, `load_employees()`, `load_project_juniors()`, `load_phases()`, при необходимости `department_load_summary()`.

**Агрегаты и логика:**

1. **KPI по статусам проектов**  
   По полю статуса проекта (в загруженных данных — `status_name` из JOIN с `project_statuses`; при отсутствии или NaN подставляется «В работе») считаются количества по значениям: «В работе», «Планируется», «Завершён», «Отменён», а также общее число проектов.

2. **Состав по проектам (таблица «Распределение сотрудников по проектам»)**  
   Для каждого проекта: ведущий — по `projects.lead_id` и имени из `employees`; рядовые — по `project_juniors` для данного `project_id` и именам из `employees`. «Всего человек» = 1 (если есть ведущий) + число рядовых по проекту.

3. **Ближайшие сроки**  
   Выбираются проекты, у которых: статус «В работе» или «Планируется»; `project_end` в интервале [сегодня, сегодня + 30 дней]. Результат — таблица (проект, дата окончания, ведущий).

4. **Проекты без полного состава**  
   Учитываются только проекты со статусом «В работе» или «Планируется». В список попадают те, у которых нет ведущего (`lead_id` пустой или NULL) или нет ни одной записи в `project_juniors` для данного проекта.

5. **Количество этапов по проектам**  
   Группировка `project_phases` по `project_id`, подсчёт числа этапов; название проекта подставляется из `projects` по `id`.

6. **Участие отделов в проектах**  
   Если у сотрудников заполнено поле `department`, для каждого участника проекта (ведущий и рядовые) берётся его отдел; строится сводка «отдел — проект — сотрудник» и при необходимости группировка по отделам и проектам для диаграммы.

**Сводка загрузки на дашборде:** для периода «сегодня — сегодня + 30 дней» вызывается `department_load_summary`; отображаются метрики ёмкости, использованных и свободных дней, «проектов можно взять» и при необходимости предупреждение о нехватке сотрудников (см. раздел 6). Блок «Аналитика и прогнозы» выводится в виде таблицы (рекомендация по набору, нехватка специалистов, перегрузки); в интерфейсе категории отображаются как «Технический специалист» и «Руководитель» (п. 6.2 — missing_juniors/missing_seniors; п. 6.3 — эквивалент недостающих из-за перегрузок). Под таблицей — expander «Прогноз: если добавить сотрудников».

## 10. Диаграмма Ганта: высота, фильтр активных, выбор проектов

**Назначение:** отображение на временной шкале выбранного набора проектов и их этапов с динамической высотой графика.

**Где используется:** раздел «Диаграмма Ганта».

**Используемые данные:** `load_projects()`, `load_phases()`; при необходимости фильтрация по статусу и по выбранным в мультиселекте проектам.

### 10.1. Высота диаграммы

- **Формула:** `gantt_height = max(400, 32 * n_rows)`, где `n_rows` — число уникальных строк на графике (каждый проект и каждый отображаемый этап — отдельная строка).
- Верхний предел высоты не задаётся: при большом числе проектов и этапов диаграмма расширяется вниз, страница прокручивается, все строки остаются читаемыми.

### 10.2. Фильтр «Только активные проекты»

- **Чекбокс** «Только активные проекты (В работе)». При включении на диаграмму попадают только проекты со статусами из настройки **«Статусы для фильтра Ганта»** (задаётся в разделе «Конфигурация»; по умолчанию «В работе»).
- **Условие:** оставляются только строки, у которых статус проекта (в загруженных данных — поле `status_name` из JOIN с `project_statuses`; при отсутствии подставляется «В работе») входит в список из конфигурации: `repository.get_gantt_active_statuses(conn)`. Для отчётов используется `repository.get_active_statuses(conn)`, для годовой ёмкости — `repository.get_capacity_statuses(conn)`. Все три списка редактируются на странице «Конфигурация».
- Реализация: `gantt_statuses = repository.get_gantt_active_statuses(conn)`; фильтрация по `df_proj['status_name'].fillna('В работе').isin(gantt_statuses)`.

### 10.3. Мультиселект проектов для отображения

- **Элемент интерфейса:** «Проекты на диаграмме (выделенные отображаются)» — мультиселект с вариантами вида «Название проекта — Статус» (по текущему `df_proj` после фильтра по активным, если он включён).
- **По умолчанию** выбраны все варианты (все проекты из текущего списка).
- **Логика:** по выбранным подписям восстанавливаются идентификаторы проектов; на диаграмму и в блок чекбоксов «Отображение этапов по проектам» попадают только проекты с `id` из выбранного множества: `df_proj = df_proj[df_proj['id'].isin(selected_ids)].copy()`.
- При пустом выборе выводится сообщение «Нет проектов для отображения.»

### 10.4. Заливка шкалы проекта по готовности

- **Назначение:** визуально показать на основной шкале проекта долю выполнения, исходя из готовности этапов (поле `completion_percent`) или из текущей даты отображения.
- **Функция:** `gantt_project_completion_percent(project_id, display_date, phases_df)`. Возвращает значение от 0.0 до 1.0.
- **Формула процента (при наличии колонки `completion_percent` у этапов):** для каждого этапа длительность в днях `d_phase = (end_date - start_date).days`; «выполнено» = `d_phase * (completion_percent / 100)` (значение 0–100). Сумма таких «выполненных» дней по всем этапам / сумма длительностей всех этапов; результат ограничен [0, 1]. Если этапов нет или сумма длительностей 0 — возвращается 0.0.
- **Формула процента (без `completion_percent`, по дате):** для каждого этапа «выполнено» к `display_date`: если `display_date >= end_date` — полные `d_phase` дней; если `start_date <= display_date < end_date` — `(display_date - start_date).days`; иначе 0. Сумма выполненных дней / сумма длительностей этапов; результат ограничен [0, 1].
- **Визуал проекта:** на той же строке, что и бар проекта, рисуется второй бар (тип «Выполнено») от `proj_start` до `completed_end = proj_start + percent * (proj_end - proj_start)`; цвет — зелёный. При проценте 0 второй бар не добавляется.
- **Визуал этапа:** для каждого этапа с `completion_percent` > 0 на той же строке этапа рисуется бар «Выполнено» от начала этапа до `phase_start + (phase_end - phase_start) * (completion_percent / 100)`.

### 10.5. Готовность этапа (completion_percent)

- **Назначение:** явно задавать процент готовности этапа (0–100 %, шаг 5) и использовать его в расчёте готовности проекта и в заливке на диаграмме Ганта.
- **Хранение:** колонка `project_phases.completion_percent` (INTEGER, по умолчанию 0). Миграция при старте приложения добавляет колонку, если её нет.
- **Интерфейс:** в разделе «Проекты и этапы» в списке этапов отображается колонка «Готовность %»; при добавлении и редактировании этапа — выпадающий список «Готовность этапа, %» с вариантами 0, 5, 10, …, 100.
- **Расчёт готовности проекта:** см. п. 10.4 (при наличии `completion_percent` используется взвешенная по длительности этапов сумма этих процентов).
- **Импорт этапов из Excel:** при вставке новых этапов в БД подставляется `completion_percent = 0`.

### 10.6. Цвета полос проектов по статусу

- **Назначение:** при отображении всех проектов на диаграмме Ганта визуально различать категории (статусы) проектов.
- **Реализация:** тип полосы проекта задаётся как «Проект (Статус)» (например, «Проект (В работе)», «Проект (Планируется)»). В `color_discrete_map` заданы цвета: **В работе** — #4caf50 (зелёный), **Планируется** — #ff9800 (оранжевый), **Завершён** — #9e9e9e (серый), **Отменён** — #bdbdbd (светло-серый). При неизвестном статусе используется «Проект» — lightgray. Полосы «Выполнено», «Обычная работа», «Командировка» сохраняют прежнюю палитру.

**Конфигурация (настройки из БД):** параметры годовой ёмкости (рабочие дни, отпуск по умолчанию, типовая длительность, доля старшего, макс. проектов, привязка ролей), плейсхолдеры дат для импорта и невалидных дат, а также флаги статусов `is_active` (отчёты), `is_capacity` (расчёт ёмкости), `is_gantt_active` (фильтр Ганта) редактируются в разделе «Конфигурация». Функции: `repository.get_config`, `repository.get_active_statuses`, `repository.get_capacity_statuses`, `repository.get_gantt_active_statuses`, `repository.get_date_placeholders`.

**Приведение типов из БД:** в `utils/type_utils.py` определены `safe_int` (в т.ч. для bytes) и `safe_date_series`; репозиторий и UI используют их для полей из БД. Имена сотрудников и проектов по id получаются точечными запросами (`get_employee_name`, `get_project_name` — один SELECT по id вместо загрузки всей таблицы).
